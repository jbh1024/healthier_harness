## Phase 3: Board + Notice + 프로필 이미지

### 3-1. DB 마이그레이션 (완료)

- **프로필 이미지**: POST/DELETE /api/users/me/profile-image (JPG/PNG, 5MB 제한)

### 3-3. Frontend (완료)
- Board: 모델(BoardPost/BoardPostDetail/Comment), Repository, Provider, BoardListScreen(고정글 아이콘), BoardDetailScreen(댓글/대댓글 UI), BoardWriteScreen
- Notice: 모델(Notice/NoticeDetail), Repository, Provider, NoticeListScreen(중요 공지 표시), NoticeDetailScreen
- 라우팅: /board, /board/write, /board/:postId, /notices, /notices/:noticeId
- HomeScreen 빠른 메뉴 모두 연결

### 의사결정
- 조회수 중복 방지: DB UNIQUE 제약 + 서비스에서 존재 여부 체크 후 INSERT → UPDATE 방식
- 대댓글 제한: 서비스 레이어에서 parent.parent != null 검증 (DB 제약 대신)
- 파일 저장소: 인터페이스 추상화로 향후 S3 전환 가능. 현재 로컬 디스크 구현체

### 5-2. Backend 수업/수강 확장 (완료)
- 수업 수정: PUT /api/academies/{id}/courses/{courseId}
- 수업 삭제: DELETE (수강 신청 없는 수업만)
- 수업 상태 변경: PUT /api/academies/{id}/courses/{courseId}/status
- 수강 취소: PUT /api/academies/{id}/enrollments/{enrollmentId}/cancel
  - APPROVED 상태 취소 시: currentEnrollment 감소 + 횟수 복구 (수업 시작일 당일이면 미복구)
  - 대기열 자동 승급: 취소 후 WAITLISTED 선두를 AUTO_APPROVE→APPROVED / MANUAL_APPROVE→PENDING으로 전환

- 게시판 수정: PUT (본인만), 삭제: DELETE (본인 또는 ACADEMY_ADMIN)
- 댓글 수정: PUT (본인만), 삭제: DELETE (본인만)
- 프로필 수정: PUT /api/users/me (이름, 전화번호)
- 비밀번호 변경: PUT /api/auth/password (현재 비밀번호 확인 후 변경)

### 5-4. Frontend (완료)
- ProfileScreen: 이름/전화번호 수정
- PasswordChangeScreen: 현재 비밀번호 확인 + 새 비밀번호 검증
- 라우팅: /profile, /settings/password

### 의사결정

- 대기열 승급: 비관적 락으로 Course 재조회 후 정원 확인 → 승급 처리

## 주요 설계 결정

| 항목 | 결정 | 근거 |
|------|------|------|
| Backend 패키지 | 도메인별 분리 (domain/{name}/entity,repo,service,controller,dto) | rules/architecture.md |
| Spring Boot 버전 | 3.4.4 (규칙 문서 3.3 → 최신 안정 버전 채택) | 하위 호환 유지 |
| RBAC | 커스텀 어노테이션 `@AcademyAuth` + AOP | 학원별 역할 검증에 적합 |
| 동시성 제어 | 비관적 락 (PESSIMISTIC_WRITE) | 정확성 우선, 단순하고 안전 |
| 알림 아키텍처 | Spring ApplicationEvent (예정) | 도메인 결합도 최소화 |
| 파일 저장소 | FileStorageService 인터페이스 추상화 | 로컬→S3 전환 가능 |
| Frontend 상태관리 | Riverpod StateNotifier + FutureProvider | README 기술스택 |
| Frontend 라우팅 | GoRouter + redirect 기반 인증/학원 가드 | 하단 탭 + 딥링크 지원 |
| API 연동 | Dio + Repository 패턴 (Retrofit 코드 생성 미사용, 수동 구현) | 초기 구현 속도 우선 |
| 학원 컨텍스트 | currentAcademyProvider 의존 → 전환 시 자동 무효화 | 멀티 학원 핵심 |
| 조회수 | DB UNIQUE + 서비스 체크 + 원자적 UPDATE | 사용자 기반 중복 방지 |
| 대댓글 | 서비스 레이어 검증 (parent.parent != null 거부) | 1depth 제한, DB 제약 불필요 |
